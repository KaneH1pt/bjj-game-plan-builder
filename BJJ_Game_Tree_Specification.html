Product Specification — Version 1.3

Personal BJJ Game Graph

Brazilian Jiu-Jitsu practitioners hold a large number of positions, transitions, and opponent reactions in their head simultaneously. Knowing what to do is only part of the challenge — understanding which parts of your game are reliable, which are developing, and which are only just being explored requires a different kind of awareness.

This tool is a personal development aid: a visual graph that maps a practitioner's own game, so they can see it, navigate it, and build it deliberately.

The graph is made up of positions (nodes) — such as Standing or De La Riva — connected by transitions (connectors between nodes), such as "pull guard, foot on hip" or "they step wide". Each connector carries a status: favourite, working on, or exploring.

Built to support the author's own BJJ practice, and to demonstrate that a domain-literate non-engineer can make rigorous architectural decisions and deliver a working product using AI as an engineering tool.

Setup & Design Decisions

This section captures the foundational decisions already made and the pre-functional work a developer must complete before any feature development begins. Each decision includes the reasoning — this is a portfolio document as much as a technical specification.

Design Decisions

Why a directed graph, not a tree

BJJ positions loop back, the same position is reachable from multiple starting points, and opponent reactions branch the path. A tree implies one direction. A graph is therefore the correct model.

Why PostgreSQL via Supabase

Hosted, managed database. Free tier covers this project indefinitely. Supabase provides authentication (verifying who a user is) and authorisation (controlling what they can access) out of the box, removing the need to manually build either mechanism. Row Level Security is built in. The visual table editor lets the author inspect and understand their own data directly.

Why FastAPI

A thin Python API layer between the frontend and database. Business logic — such as preventing deletion of a node that still has connectors attached — needs a trusted layer to enforce it. Python is readable and AI-generatable. Maps naturally to data engineering mental models the author already holds from healthcare interoperability and data transformation work.

Why React

The application has significant interactive state — node selection, filter toggles, panel open/close, in-browser editing. Plain HTML is unmaintainable at this level of interactivity. React is the industry standard for this class of problem.

Why Vercel + Railway

Both connect directly to GitHub and auto-deploy on commit. A developer pushes a change, it is live within 60 seconds. Zero manual deployment steps once connected.

Why BDD for this specification

Scenarios written in Behaviour Driven Development format are readable by a non-technical stakeholder, a developer, and a tester equally. They drive development, testing, and documentation from a single source. Each scenario is a thin vertical slice — the smallest demonstrable unit of working software running through all layers.

On the MVP

The working MVP (bjj-game-plan-builder.html) is a single file with data hardcoded as JSON. It proves the concept works and is the visual and behavioural reference for all frontend development.

Why closed beta for initial launch — no public signup

An open signup page is the highest-risk surface in the application — it is the one endpoint anyone on the internet can hit without credentials. Even with CAPTCHA and rate limiting, it is an attack vector a closed beta does not need. Accounts are created manually by the product owner via the Supabase dashboard. Trusted testers log straight in. The signup scenarios are written and ready to build — they are deferred until the app is stable and the security posture is validated, not discarded.

Why not start with a walking skeleton including account creation

A walking skeleton with an open signup page would expose the highest-risk endpoint before any other part of the application is hardened. Starting with login-only for a known set of manually created accounts removes that surface entirely for the closed beta phase. The signup flow is specified and will be built — it follows the Edit & Delete milestone when the application is stable enough to open.

Why three agent tiers with different models

The dev agent uses Opus — it performs the most complex task (reading a specification and writing conformant code across multiple files and layers). The review agent uses Sonnet — it reads code and produces a compliance report, a reasoning-heavy but less generative task. The test agent uses Haiku — it runs BDD scenarios mechanically against a known interface, a high-volume, low-complexity task. Matching model capability to task complexity controls cost without sacrificing quality where it matters.

Agent Pipeline

This project is built using an AI agent pipeline. Agents are configured as GitHub Actions workflows living in the repository. The product owner (Kane Hellewell, author of this document) owns the specification — agents own implementation.

The pipeline calls the Anthropic API directly via Claude Code. Claude Code is the tool that reads files, writes code, and interacts with the repository. The LLM model used for each call is specified per agent — model selection is a deliberate cost and quality decision, not a default.

Roles and Responsibilities

  • Dev Agent: Reads the specification and writes code to implement the specified feature. Raises a pull request when complete. Does not run tests, does not merge its own pull request, does not modify the specification. If it cannot implement something as specified, it leaves a comment on the pull request explaining why — it does not change the spec to fit what it could build.
  • Test Agent: Runs the BDD scenarios against the pull request. Reports which scenarios pass and which fail, with the scenario ID (e.g. AUTH-03). Does not fix code. Does not modify scenarios. Blocks merge if any scenario fails. If a scenario cannot be run, it flags this explicitly rather than skipping silently.
  • Review Agent: Reads the pull request and the specification. Produces a plain English report stating whether the implementation matches the spec. Flags anything in the code that contradicts a requirement, even if tests pass. Does not fix code, does not approve or merge. References requirements by their identifier (e.g. SEC-04, NOD-03) in its report.
  • Product Owner (Kane Hellewell): Only person who can merge a pull request. Owns the specification — any change to requirements must come from the product owner, not an agent. Reviews the test report and review report before merging. Can reject a pull request and reopen the issue for the dev agent to rework.

Key principle — incremental, feature by feature: Agents are triggered one feature at a time, following the specification order: Authentication → Nodes → Connectors. Each feature is built, tested, and reviewed before the next begins.

The order within each feature is fixed:

  • Product owner raises a GitHub Issue labelled build for the feature
  • Dev Agent writes the code and raises a pull request
  • Test Agent runs — if any scenario fails, it blocks the pull request and the dev agent must rework
  • Review Agent fires only after all tests pass — it reviews code that works, not broken code
  • Product owner reads both reports and merges if satisfied, or rejects and reopens the issue

Cost safeguarding: Agents only trigger on issues and pull requests the product owner has explicitly labelled. One agent run per feature. A hard token budget is set in each agent prompt. Supabase and Railway both hard-stop at free tier limits rather than charging overages.

Security: GITHUB_TOKEN scoped to the repository, expires at end of each run. All credentials stored as GitHub Secrets — encrypted, never visible in code. Only the product owner (Kane Hellewell) can merge.

Technical Notes

  • Authentication provider: Supabase Auth. Username and password only.
  • Closed beta account creation: Accounts are created manually by the product owner in the Supabase dashboard. No signup form is built in this milestone.
  • Username validation: 3–30 characters. Letters, numbers, underscores, and hyphens only — no spaces or other special characters. Case insensitive. Must be unique.
  • Password validation: Minimum 10 characters. Must contain at least one uppercase letter, one lowercase letter, and one number. No maximum length — password managers must be supported. Never stored in plain text — hashed by Supabase before reaching the database.
  • Token handling: On login, Supabase returns a JWT. The frontend includes it in every API call header. The backend validates it and extracts the user ID to scope all data queries.
  • Token storage: Memory or secure httpOnly cookie. Never localStorage.
  • Idle timeout: 30 minutes of inactivity triggers automatic logout and redirect to /login.
  • Row Level Security: Enabled on all tables. Policies enforce that a user can only read and write their own rows — at the database layer, not just the application layer.
  • Error messages: All auth errors return "Incorrect username or password" regardless of which field is wrong. This prevents an attacker using the error to determine whether a username exists.

Security

Security requirements expressed as functional statements — clear, testable rules for a developer or agent. The tooling handles most of the implementation; this section defines what must be true.

  • SEC-01 — HTTPS everywhere: All traffic is encrypted in transit. Vercel and Railway enforce HTTPS automatically. HTTP requests must be rejected.
  • SEC-02 — Authentication required on all data routes: Every endpoint except login requires a valid JWT. Unauthenticated requests return 401. The backend validates the token on every request — not just on login.
  • SEC-03 — Data ownership at the database layer: Row Level Security is enabled on all tables. Policies enforce that a user can only read and write rows where user_id matches their authenticated identity — regardless of what the application layer does.
  • SEC-04 — Tokens stored safely: JWTs stored in memory or a secure httpOnly cookie. Never in localStorage, where they are accessible to JavaScript and vulnerable to cross-site scripting attacks.
  • SEC-05 — CORS locked to the frontend origin: The backend only accepts requests from the Vercel frontend URL. Requests from any other origin are rejected.
  • SEC-06 — No secrets in code: All credentials stored as environment variables. Nothing sensitive is hardcoded or committed to the repository.
  • SEC-07 — Nothing sensitive in URLs: User identifiers, tokens, and data must not appear in URL paths or query strings.
  • SEC-08 — General rate limiting: 60 requests per authenticated user per minute. The dev agent implements rate limiting middleware in FastAPI, reading an ENVIRONMENT variable to apply the correct limit. The product owner configures the ENVIRONMENT variable in the Railway dashboard — production for the prod service, test for the test service.
  • SEC-09 — Signup rate limiting by IP: Maximum 5 account creation attempts per IP address per hour. Deferred to Public Signup milestone.
  • SEC-10 — CAPTCHA on signup: hCaptcha or Cloudflare Turnstile required on the signup form. Deferred to Public Signup milestone.
  • SEC-11 — Input length limits: Labels capped at 100 characters. Notes capped at 5,000 characters. Oversized payloads are rejected before reaching the database.
  • SEC-12 — Dependency monitoring: Dependabot enabled on the GitHub repository. Raises a pull request automatically when a dependency has a known vulnerability.
  • SEC-14 — Parameterised queries only: All database queries use parameterised statements. Raw string interpolation with user-supplied input is forbidden. This applies to every query written by the dev agent — no exceptions. This prevents SQL injection regardless of what input reaches the database layer.
  • SEC-15 — Security logging: Failed login attempts and invalid access attempts on authenticated routes are logged with a timestamp and IP address. Logging is implemented as middleware in the FastAPI backend and stored in Railway. Logs are retained for 30 days. This is the minimum needed to detect brute force patterns. Personal data is not logged.
  • SEC-16 — No verbose errors in production: The production environment must never return stack traces, internal error messages, or framework debug output to the client. All unhandled errors return a generic 500 Internal Server Error response. Verbose error detail is permitted in the test environment only.

Appendix B — Glossary

  • API (Application Programming Interface): A contract that defines how two systems communicate. Here, the FastAPI backend exposes an API that the React frontend calls to read and write data.
  • BDD (Behaviour Driven Development): A specification approach that writes requirements as user behaviours in plain language. Readable by non-technical stakeholders, developers, and testers equally. Each scenario is independently testable.
  • Connector: A directed line between two nodes representing what happens between positions — triggered by either the user's action or the opponent's reaction. Has a status of favourite, working on, or exploring. Referred to as an "edge" in the database and API layer, where it is a standard graph term.
  • JWT (JSON Web Token): A standard encoded string used to prove a user's identity to the API without sending credentials on every request. Issued by Supabase on login.
  • Node: A position in the BJJ game graph. Examples: Standing, De La Riva, Side Control Top.
  • RLS (Row Level Security): A database feature that enforces data ownership at the database layer. Policies ensure a user can only read and write their own rows regardless of what the application layer does.
  • Status: A classification on a connector. favourite = reliable, used regularly. working on = actively developing. exploring = attempted but not yet understood.
  • Vertical slice: The smallest possible strip of working software running through all layers — database, API, frontend. Each feature in this specification is a vertical slice.
